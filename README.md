### 렌더링 방식 탐구 및 비교 분석

- SPA : 하나의 페이지로 이루어진 앱. 페이지를 들어가면 HTML 파일을 다운로드하고, 페이지가 나누어져 있지 않기 때문에 렌더링할 때 오래 걸린다.
- SSR : 페이지를 들어가면 서버에서 HTML 파일을 생성하고, 생성한 파일을 렌더링하기 시작한다.
- SSG : 페이지를 들어가면 이미 빌드시 만들어진 HTML 파일을 렌더링한다.
- ISR : SSG지만 revalidate를 통해 데이터를 업데이트 할 수 있다. revalidate 시간이 지나면 재빌드하여 HTML 파일을 생성하고 렌더링한다.

사용자 경험(UX) 측면 분석: 각 렌더링 방식이 사용자 경험에 어떻게 영향을 미치는지 설명하고, 본인의 서비스를 사례로 들어 실제 적용된 예를 보여줍니다.

SPA를 사용하면 다른 페이지로 이동할 때 깜빡거리지 않고 부드럽게 넘어간다. SSR, SSG, ISR을 사용하면 렌더링 속도가 CSR보다 빠르다. 하지만 SSG를 사용하면 데이터가 업데이트 됐을 때, 업데이트된 정보가 안 보인다. 빌드됐을 때 만들어진 HTML 파일을 렌더링하기 때문이다. 따라서 동적인 파일은 SSR을 사용하고, 정적인 파일인 경우에만 SSG를 사용하는 것이 좋다. ISR은 revalidate를 이용해 SSG의 단점을 보완할 수 있다.

카페인 서비스 클론 코딩에서는 초기 메인 페이지를 렌더링할 때 SSG을 사용했습니다. 실제 카페인 서비스에 들어가면 지도 요청 상태가 SUCCESS로 변경되어야만 로딩중이 아닌 다른 ui도 보이기 시작합니다.
